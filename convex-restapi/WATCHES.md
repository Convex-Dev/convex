# Convex MCP State Watches — Design

## Context

Convex MCP exposes on-chain state via tools over the MCP Streamable HTTP transport. We want clients to **watch** paths in the CVM global state and receive notifications when values change.

This document describes how state watches work with MCP Streamable HTTP on Convex peers, following the [MCP specification (2025-11-25)](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports#streamable-http).

## Relevant MCP Spec Rules

### POST /mcp (client → server requests)

- Client MUST include `Accept: application/json, text/event-stream`.
- Server MUST respond with either `application/json` or `text/event-stream`.
- If SSE: server sends response(s) as SSE events.
- **"After all JSON-RPC responses have been sent, the server SHOULD close the SSE stream."** — POST SSE streams are short-lived, per-request.

### GET /mcp (server → client notifications)

- Client MAY open a GET SSE stream for server-initiated messages.
- Messages on this stream **"SHOULD be unrelated to any concurrently-running JSON-RPC request"** — this is the spec-defined channel for ongoing notifications like watch changes.
- **"The server MAY close the SSE stream at any time. The client MAY close the SSE stream at any time."** — both sides accept the stream is ephemeral.
- Server MAY return 405 if it doesn't support GET streams.

### Session management

- Server **MAY** assign session ID on `initialize` (via `Mcp-Session-Id` response header).
- Client **MUST** include session ID on all subsequent requests.
- Server **MAY** terminate session at any time → responds 404.
- Client **SHOULD** send DELETE to terminate when done.

## Design

### Lifecycle

```
1. POST initialize
   → Server returns Mcp-Session-Id header (random UUID)
   → If client accepts SSE: respond with SSE (send result, close stream per spec SHOULD)
   → If client accepts JSON only: respond with JSON
   → NO server-side state created (session ID is not stored)

2. GET /mcp  (Mcp-Session-Id: <id>, Accept: text/event-stream)
   → Server creates McpConnection keyed by session ID
   → Opens long-lived SSE stream with keepalives
   → This is the notification delivery channel for watches
   → This is the only point where server-side state is created

3. POST tools/call watchState  (Mcp-Session-Id: <id>)
   → Server looks up McpConnection by session ID
   → If not found: tool error ("watchState requires an active GET /mcp stream")
   → Adds WatchEntry to the McpConnection
   → Starts watcher if not running
   → Response: JSON with {watchId: "w-1", value: ...}

4. ConvexStateWatcher polls CVM state
   → On change: sends JSON-RPC notification on the McpConnection's GET stream
   → {"jsonrpc":"2.0","method":"notifications/stateChanged","params":{...}}

5. POST tools/call unwatchState  (Mcp-Session-Id: <id>)
   → Finds and removes watch by watchId or path prefix
   → Response: JSON with {removed: N}

6. Client disconnects GET stream (or network drops)
   → McpConnection destroyed
   → All watches on the connection destroyed
   → Session entry removed from map
   → Zero server-side state remains

7. DELETE /mcp  (Mcp-Session-Id: <id>)
   → Same cleanup as disconnect: close connection, destroy watches
   → Returns 200
```

### Why GET /mcp for watches (not POST SSE)

The spec says POST SSE streams "SHOULD" be closed after responses are sent. POST SSE is designed for streaming a single response (e.g., a long-running tool result), not for ongoing notifications.

GET /mcp is explicitly designed for server-initiated messages "unrelated to any concurrently-running request" — exactly what watch change notifications are.

This also means watches are naturally ephemeral: the client opens a GET stream, registers watches, receives notifications, and when the stream drops (for any reason), watches die. It is the **client's responsibility** to reconnect and re-register watches if desired.

### Why no state on initialize

A flood of `initialize` calls creates zero server-side resources. The session ID is returned but not stored. The only heavy resource (McpConnection: virtual thread + TCP socket) is created on GET /mcp, which is bounded by `MAX_CONNECTIONS`.

The session ID serves purely as a correlation token — linking POST requests to the right GET stream. Since it's generated by the server (random UUID), collisions are not a practical concern. A client that sends POST watchState with a bogus session ID simply gets "no active connection" — no harm done.

## Architecture

```
Client                              Convex Peer (McpAPI)
  │                                      │
  │── POST initialize ──────────────────>│  Returns Mcp-Session-Id: abc123
  │<── SSE or JSON: InitializeResult ───│  NO state created
  │                                      │
  │── GET /mcp ─────────────────────────>│  Creates McpConnection keyed by "abc123"
  │    Mcp-Session-Id: abc123            │  Opens SSE stream
  │    Accept: text/event-stream         │

  │    : keepalive                       │
  │                                      │
  │── POST tools/call watchState ───────>│  Finds McpConnection for "abc123"
  │    Mcp-Session-Id: abc123            │  Adds watch to connection
  │<── JSON: {watchId: "w-1"} ──────────│
  │                                      │

  │                                      │
  │<── GET stream SSE event: ────────────│  Notification on GET stream
```

## Components

### McpConnection

Wraps a response `PrintWriter` for the GET /mcp stream. Owns watches.

```java
public class McpConnection {
    final PrintWriter writer;
    volatile boolean closed = false;

    // Watches owned by this connection
    final ConcurrentHashMap<String, WatchEntry> watches = new ConcurrentHashMap<>();
    final AtomicLong watchCounter = new AtomicLong(0);

    void sendEvent(String eventType, String data);
    String addWatch(ACell[] path, AVector<ACell> pathVec, String pathString, Hash initialHash);
    boolean removeWatch(String watchId);
    long removeWatchesByPathPrefix(AVector<ACell> prefix);
    boolean hasWatches();
    void close();
    boolean isClosed();
}
```

When `close()` is called, the connection is marked closed and removed from the map. Watches are implicitly destroyed (GC'd with the connection).

### McpAPI changes

```java
// Connection map: session ID → McpConnection (created on GET /mcp)
private final ConcurrentHashMap<String, McpConnection> connections = new ConcurrentHashMap<>();

// Watcher (Convex-specific, reads CVM state directly)
private final ConvexStateWatcher stateWatcher;
```

**handleMcpRequest (POST):**
- `initialize` → returns session ID in header, no map entry created. Responds with SSE or JSON per client Accept header. If SSE, sends result as event and closes stream.
- `watchState` → gets session ID from header, looks up McpConnection in map, adds watch. Responds with JSON.
- Other tools → unchanged (JSON response, stateless).

**handleMcpGet (GET):**
- Validates `Accept: text/event-stream`.
- Gets session ID from `Mcp-Session-Id` header. If missing, generates one and returns in response header.
- Enforces `MAX_CONNECTIONS`.
- Creates McpConnection, adds to map keyed by session ID.
- Opens SSE stream with keepalive loop.
- On disconnect: removes from map, close connection (watches die).

**handleMcpDelete (DELETE):**
- Gets session ID from header.
- Removes McpConnection from map, closes it.
- Returns 200 (or 404 if not found).

### ConvexStateWatcher

Convex-specific inner class of McpAPI. Polls CVM global state and pushes notifications to McpConnections.

```java
class ConvexStateWatcher {
    static final long POLL_INTERVAL_MS = 1000;
    static final long VALUE_SIZE_THRESHOLD = 1024;

    private volatile Thread thread;
    private volatile boolean running;

    void ensureRunning();   // start watcher thread if not running
    void shutdown();        // stop watcher thread

    private void pollLoop() {
        while (running && hasAnyWatches()) {
            for (McpConnection conn : connections.values()) {
                if (conn.isClosed() || !conn.hasWatches()) continue;
                for (WatchEntry entry : conn.watches.values()) {
                    ACell value = RT.getIn(server.getState(), entry.path);
                    Hash hash = Hash.get(value);
                    if (!hash.equals(entry.lastHash)) {
                        entry.lastHash = hash;
                        conn.sendEvent("message", buildNotification(entry, value));
                    }
                }
            }
            Thread.sleep(POLL_INTERVAL_MS);
        }
    }
}
```

Daemon virtual thread. Starts on first watch, exits when no watches remain.

## Resource Limits

| Limit | Value | Rationale |
|-------|-------|-----------|
| `MAX_CONNECTIONS` | 1000 | Each holds a virtual thread + TCP socket |
| `MAX_WATCHES_PER_CONNECTION` | 16 | Caps polling overhead per client |
| `MAX_BATCH_SIZE` | 20 | Limits per-request processing |

## DoS Considerations

| Attack | Mitigation |
|--------|-----------|
| Flood `initialize` | No state created. Zero cost. |
| Flood GET /mcp | `MAX_CONNECTIONS` hard cap. Each is a real TCP socket so OS limits also apply. |
| Flood `watchState` | Requires valid McpConnection (must hold a GET stream). `MAX_WATCHES_PER_CONNECTION` caps per-client. |
| Open connection, never watch | Connection holds one virtual thread + socket. Bounded by `MAX_CONNECTIONS`. No polling cost. |
| Watch expensive paths | Polling cost is per-watch, bounded by total watches across all connections. |
| Slow client (backpressure) | `sendEvent` is synchronised; writer error → connection closed. |

**High-value peers (large stake, critical infrastructure) should disable MCP entirely** and leave it to lower-staked proxy/gateway peers.

## Client Responsibilities

- **Reconnect and re-register.** Watches are ephemeral. If the GET stream drops (network, server restart, timeout), all watches are lost. The client must reconnect (new `initialize` → new GET /mcp) and re-register watches.
- **Accept loss.** Notifications are best-effort. If the connection drops between polls, the client may miss a change. The client should poll current state on reconnect.
- **Clean up.** Client SHOULD send DELETE when leaving, per MCP spec. But the server does not depend on this — disconnect cleans up regardless.

## Streamable HTTP in Javalin/Jetty

Convex MCP runs on **Javalin 6** backed by **Jetty 12**. Here is how MCP Streamable HTTP maps to the Javalin/Jetty stack.

### Route registration

```java
app.post("/mcp", this::handleMcpRequest);
app.get("/mcp",  this::handleMcpGet);
app.delete("/mcp", this::handleMcpDelete);
```

All three verbs share a single `/mcp` path — standard Javalin routing.

### POST /mcp — request/response (with optional SSE)

Javalin provides a `Context` wrapping the servlet request/response. For normal JSON responses use `ctx.result(json)` — Javalin writes the body, sets `Content-Type`, and closes the response automatically.

For **SSE responses on POST** (required by MCP when the client sends `Accept: text/event-stream` without `application/json`), Javalin's normal result mechanism cannot be used because it would override the content type. Instead we **bypass Javalin and write directly to the servlet response**:

```java
HttpServletResponse res = ctx.res();      // access raw servlet response
res.setStatus(200);
res.setContentType("text/event-stream");
res.setCharacterEncoding("UTF-8");
res.setHeader("Cache-Control", "no-cache");
PrintWriter writer = res.getWriter();
writer.write("event: message\ndata: {json}\n\n");
writer.flush();
res.flushBuffer();                         // COMMITS the response
```

**`res.flushBuffer()`** is the critical call — it commits the HTTP response headers to the wire. After this, Javalin's after-handlers and result writers cannot modify the status, headers, or body. The POST handler then returns normally. This is a short-lived SSE stream per MCP spec — a single event is sent and the stream closes when the handler returns.

### GET /mcp — long-lived SSE stream

The GET handler opens a long-lived SSE connection by blocking a virtual thread:

```java
HttpServletResponse res = ctx.res();
res.setContentType("text/event-stream");
res.setCharacterEncoding("UTF-8");
res.setHeader("Cache-Control", "no-cache");
res.setHeader("X-Accel-Buffering", "no");  // disable nginx buffering
res.flushBuffer();                          // commit response headers

PrintWriter writer = res.getWriter();
McpConnection conn = new McpConnection(writer);
connections.put(sessionId, conn);

try {
    while (!conn.isClosed()) {
        writer.write(": keepalive\n\n");   // SSE comment (no event fired)
        writer.flush();
        if (writer.checkError()) break;    // client disconnected
        Thread.sleep(30_000);              // yield virtual thread
    }
} finally {
    conn.close();
    connections.remove(sessionId);
}
```

**Key behaviours:**

- **Javalin/Jetty keeps the connection alive** as long as the handler thread is running. The handler blocks in the keepalive loop; Jetty does not time out the request because the handler thread is still active.
- **Virtual threads** — Jetty 12 (used by Javalin 6) runs request handlers on virtual threads. Each SSE connection blocks a virtual thread (not a platform thread), so thousands of concurrent SSE connections are feasible without thread pool exhaustion. `Thread.sleep()` in the loop yields the virtual thread to the carrier.
- **Disconnect detection** — `writer.checkError()` returns `true` when the client TCP connection drops (Jetty sets the error flag on the underlying `ServletOutputStream`). The handler breaks out of the loop and cleans up in `finally`.
- **Notifications** — The `ConvexStateWatcher` thread calls `conn.sendEvent()` from a different thread. The `sendEvent()` method is `synchronized` on the same `PrintWriter` to prevent interleaving with keepalive writes.

### Session header

- **Reading:** `ctx.header("Mcp-Session-Id")` extracts the request header.
- **Writing:** `ctx.res().setHeader("Mcp-Session-Id", id)` sets the response header via the servlet API. This must be done before `flushBuffer()` commits the response.

### DELETE /mcp — session termination

Standard Javalin handler. Looks up and destroys the McpConnection by session ID, returns `200` or `404`. No special servlet handling needed.

### Reverse proxy considerations

- **`X-Accel-Buffering: no`** — Tells nginx not to buffer the SSE stream.
- **`Cache-Control: no-cache`** — Prevents intermediary caching of SSE responses.
- Reverse proxies may impose their own idle timeouts. The 30-second keepalive interval is chosen to stay below common proxy timeouts (typically 60s).

## Shared Infrastructure

- `SseConnection` — original shared class remains for Covia's use (thin PrintWriter wrapper).
- `McpSession` — remains available for Covia (persistent sessions across reconnects, multiple connections per session).
- `StateWatcher` — generic shared class remains for Covia (pluggable StateResolver).
- Convex uses its own `McpConnection` and `ConvexStateWatcher` — Convex-specific, reads CVM state directly.
